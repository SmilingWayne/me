# Dijkstra 算法的桶优化：Dial Algorithm & Radix Heap

!!! note "Conclusion"
    Dijkstra算法是求解单源最短路径问题的经典算法，其核心操作包括：**找出当前距离最小的节点**（Extract Min）和**更新邻接节点的距离**（Decrease Key）。
    
    在稀疏图上，使用二叉堆或斐波那契堆可以得到较好的复杂度，但在某些特殊场景下，利用**桶（Bucket）结构**可以进一步加速，特别是当边权为小整数时。本文介绍两种基于桶的Dijkstra变体：**Dial算法（简单桶）** 和 **Radix Heap（基数堆）**，并对比其原理、实现细节、优缺点。

---

## 1. Dial算法（Simple Buckets / Dial’s Algorithm）

### 1.1 原理

Dial算法利用一组连续编号的桶，每个桶代表一个**距离值**（桶宽度为1）。节点的当前距离标签 `d[v]` 决定它属于哪个桶。初始时，源点距离为0，放入桶0；其他节点距离为∞，不放入任何桶。算法重复以下步骤：

- **Find Min**：从最小的桶索引开始向右扫描，找到第一个非空桶，该桶中的节点即为当前距离最小的节点（可能有多个，任取一个）。
- **Relaxation**：对该节点进行松弛操作，更新其邻接节点的距离。若某个邻接节点的距离减小，则将其从原桶移动到新桶（如果原桶存在）。

由于桶的索引对应距离值，扫描时需要从当前最小距离位置开始。为了处理距离不断增大的情况，通常使用**循环桶（circular buckets）**，因为距离可能超过最大桶数，但实际可能的最大距离受限于图中最大边权乘以节点数，或者我们可以动态调整。

### 1.2 伪代码

```
function Dial(G, s, w_max):
    n = |V|
    C = (n-1) * w_max  // 最大可能距离
    buckets = array of lists of size C+1
    pos = array of (bucket_index, node_ptr) for each node
    dist = [∞] * n
    dist[s] = 0
    buckets[0].append(s); pos[s] = (0, node_ptr)
    cur = 0
    while not all nodes permanent:
        while buckets[cur] is empty:
            cur = (cur + 1) % (C+1)
        u = buckets[cur].pop_front()
        pos[u] = None
        // relax
        for v in Adj[u]:
            new_dist = dist[u] + w(u,v)
            if new_dist < dist[v]:
                old = dist[v]
                dist[v] = new_dist
                if old != ∞:
                    remove v from buckets[old] using pos[v]
                insert v into buckets[new_dist] and update pos[v]
                // 如果 new_dist < cur，可以重置 cur = new_dist？但通常不必，因为扫描会继续
    return dist
```

### 1.3 细节与实现要点

- **桶的数量**：需要知道距离的上界。若边权为整数且最大值为 `w_max`，节点数为 `n`，则最短路径长度不超过 `(n-1)*w_max`。但实际中常采用循环桶，桶数取 `w_max+1` 或稍大，并利用取模运算循环使用。
- **桶内数据结构**：每个桶可以用链表或动态数组存储节点。由于需要删除节点（当距离更新时），最好使用双向链表以便 O(1) 删除。
- **扫描指针**：每次找最小桶时，从上次的指针开始扫描，但新插入的节点可能距离更小（比如负权？不，Dijkstra要求非负），因此实际上距离只会增大，所以指针可以单调递增，直到循环。但使用循环桶后，当指针超过最大桶时取模，此时要小心处理：因为新距离可能小于当前指针（在循环意义上），但实际上由于非负权，新距离不可能小于已经永久标记的节点距离，所以指针单调递增（不考虑循环）也是可行的，只是需要处理循环边界。

### 1.4 优势

- **实现简单**：桶数组直观，操作简单。
- **在小整数权图上极快**：如果边权范围小，桶数量少，每次Find Min几乎为 O(1)，因为只需扫描少量桶。
- **适合稠密图？** 不，主要受益于权值小。

### 1.5 缺陷
- **空间开销大**：需要与最大距离成正比的桶数量，当权值大或节点多时，内存无法承受。
- **扫描开销可能大**：如果桶很多但稀疏，Find Min可能需要扫描大量空桶。
- **无法处理大权值**：距离可能非常大，导致桶数量巨大。
- **删除操作需要维护链表**：实现稍复杂，且需要快速定位节点在桶中的位置（通常用数组记录每个节点所在的桶和链表节点）。

## 2. Radix Heap（基数堆）

### 2.1 原理

Radix Heap 是对 Dial 算法的改进，它使用**指数增长的桶宽度**，每个桶覆盖一个距离区间，而不是单一距离值。桶的数量通常很少（例如 O(log C)），其中 C 是最大距离。每个桶有一个范围 [low, high]，且这些范围是不重叠且连续的，从低到高覆盖所有可能距离。初始时，桶的范围可能设置为：`[0,0]`, `[1,1]`, `[2,3]`, `[4,7]`, `[8,15]`, ... 即第 k 个桶的宽度为 2^k（从某个起点开始）。

算法过程：

- **Insert**：节点根据其距离放入对应的桶（距离落在哪个区间就放入哪个桶）。
- **Find Min**：找到最小的非空桶。如果该桶的宽度为 1（即只有一个可能距离）或者桶内只有一个节点，则直接取出该节点作为最小节点。否则，需要**重新分配（redistribute）** 这个桶的范围：将该桶的区间细分成更小的子区间（通常按指数分裂），然后将桶内所有节点重新分配到这些子桶中，直到最小桶宽度为1或只有一个节点。这个过程会创建新的桶（或重用现有桶），但总桶数仍然有限。

### 2.2 伪代码

```
初始化桶区间，例如：
    bucket_ranges = [ [0,0], [1,1], [2,3], [4,7], [8,15], [16,31], ... ] 直到覆盖最大距离
    buckets = list of lists 对应每个区间
    每个节点记录所在桶索引
    dist 数组

while 有节点未处理:
    // 找最小非空桶
    i = 0
    while i < len(buckets) and buckets[i] is empty:
        i++
    // 如果桶宽度>1且节点数>1，则重新分配
    while bucket_ranges[i][1] - bucket_ranges[i][0] > 0 and len(buckets[i]) > 1:
        // 分裂该桶
        L, R = bucket_ranges[i]
        // 确定分裂方式：例如将区间分成多个子区间，每个子区间宽度为 2^(k-1) 类似初始
        // 这里简化：按2的幂次分裂，例如 [L, L], [L+1, L+1], [L+2, L+3], [L+4, L+7], ... 直到覆盖 R
        new_ranges = []
        start = L
        while start <= R:
            width = 1
            while start + width*2 - 1 <= R and width < something: // 实际上按指数增长
                width *= 2
            // 但为了简单，可以直接用 [start, min(start+width-1, R)]
            // 更规范做法：保持与初始划分一致，即每个子区间长度是2的幂次，且最小为1
            // 实现略
        // 创建新桶，将原桶中节点移到新桶，更新桶列表
        // 然后重新开始找最小桶
        i = 0
        while buckets[i] is empty:
            i++
    // 现在桶i只有一个节点或区间宽度为1
    u = buckets[i].pop()
    // 标记永久
    // 松弛...
```

### 2.3 细节与实现要点

- **桶的范围设计**：通常取 2 的幂次递增。例如：桶0: [0,0]，桶1: [1,1]，桶2: [2,3]，桶3: [4,7]，桶4: [8,15]，... 直到覆盖最大可能距离。注意这些区间是连续的且不重叠，并且每个区间的长度是 2^(k-1) 对于 k≥2。
- **重新分配过程**：当最小桶包含多个节点且区间宽度 >1 时，需要将该桶的区间细分为更小的子区间，并将节点重新分配。例如桶 [2,3] 可以分成 [2,2] 和 [3,3] 两个桶。如果区间更大，如 [8,15]，可以分成 [8,8], [9,9], [10,11], [12,15] 等，但通常采用与初始划分一致的指数分裂方式，比如再次按照 2 的幂次分裂。分裂后，原来的桶被新桶取代，新桶的区间范围更新，且索引可能重新排列。
- **节点移动**：重新分配时，需要遍历该桶的所有节点，根据新距离放入对应的新桶。由于桶的区间是连续的，每个节点的新桶可由距离直接确定。
- **删除操作**：类似于 Dial 算法，更新距离时需要从旧桶删除节点，插入新桶。这要求每个节点知道自己在哪个桶中，并且桶内支持快速删除（如双向链表）。

### 2.4 优势

- **桶数量少**：约为 O(log C)，C 为最大距离，空间效率高。
- **Find Min 操作高效**：虽然可能需要多次重新分配，但每个节点每次重新分配后进入更精细的桶，且每个节点被重新分配的次数有限（O(log C)）。总复杂度接近 O(m + n log C)，在整数权下优于二叉堆。
- **适合大范围整数权**：即使距离很大，只要范围在可表示的整数内，桶数量可控。

### 2.5 缺陷

- **实现复杂**：需要动态管理桶的区间和节点，重新分配逻辑较复杂。
- **重新分配开销**：当最小桶包含大量节点时，重新分配需要遍历所有节点，可能产生较大常数。
- **仍需要处理删除操作**：节点更新时需从链表中移除，同样需要双向链表。

## 3. 对比总结

| 特性             | Dial算法 (Simple Buckets)                    | Radix Heap                             |
| ---------------- | -------------------------------------------- | -------------------------------------- |
| **桶结构**       | 每个桶对应一个距离值（宽度1）                | 每个桶对应一个距离区间，宽度指数增长   |
| **桶数量**       | 与最大距离成正比，可能非常大                 | 与最大距离的对数成正比，通常很少       |
| **Find Min操作** | 扫描空桶，可能耗时                           | 找到最小非空桶，若桶内多节点则重新分配 |
| **节点移动**     | 每次更新距离需移动节点到新桶                 | 同样需移动，但重新分配时可能批量移动   |
| **空间复杂度**   | O(C + n)，C为最大距离                        | O(log C + n)                           |
| **时间复杂度**   | O(m + n*C) 最坏，实际若边权小则近似 O(m + n) | O(m + n log C) 理论，实际常数稍大      |
| **适用场景**     | 边权为小整数（如≤100），距离范围有限         | 边权为中等整数，范围较大但可表示       |
| **实现难度**     | 较简单                                       | 较复杂，需处理桶分裂                   |
| **典型应用**     | 交通网络（小权值），通信网络等               | 一般整数权图，如路由算法               |

### 3.1 举例对比

假设最大距离为 1000：

- Dial算法需要 1001 个桶，Find Min 可能扫描很多空桶。
- Radix Heap 只需约 log2(1000) ≈ 10 个桶，但需要维护区间。

如果边权都是 1 或 2，Dial算法非常高效，因为桶很少且扫描几乎无空桶。

### 3.2 共同点

- 都利用了桶来组织节点，避免了堆的 percolate 操作。
- 都需要支持节点的删除和插入，通常用双向链表实现桶内节点。
- 都要求边权为非负整数。
